# Robot-World / Hand-Eye (AX=ZB) 캘리브레이션 강의 노트 — 6DOF 로봇 + MuJoCo 시뮬레이션

체커보드를 로봇 말단(플랜지)에 부착하고 **외부에 고정된 카메라**로 여러 자세를
촬영해, 말단→보드 포즈 $X$와 베이스→카메라 포즈 $Z$를 **동시에** 추정하는
**robot-world/hand-eye (AX=ZB, 문헌에 따라 AX=YB) 캘리브레이션**을 MuJoCo
시뮬레이션으로 구현한다. MG400 실습(`../handeye_calibration.py`)과 같은
eye-to-hand 구성이지만, AX=XB + 트릭 대신 AX=ZB 폐형해로 미지수 두 개를 한 번에 푼다.

## 0. 표기법

$^{a}T_{b} \in SE(3)$ : 좌표계 $b$의 점을 좌표계 $a$로 옮기는 동차변환.
합성 규칙: $^{a}T_{c} = {^{a}T_{b}}\,{^{b}T_{c}}$.

| 기호 | 의미 | 어떻게 아는가 |
|---|---|---|
| $A_i = {^{b}T_{g,i}}$ | 자세 $i$에서 베이스→그리퍼(플랜지) | 로봇 FK (엔코더) — **알려짐** |
| $B_i = {^{c}T_{t,i}}$ | 자세 $i$에서 카메라→보드(target) | 체커보드 PnP — **측정됨** |
| $X = {^{g}T_{t}}$ | 그리퍼→보드 (부착 포즈) | **미지수 1** |
| $Z = {^{b}T_{c}}$ | 베이스→카메라 | **미지수 2** |

## 1. 문제 설정과 관계식

보드는 그리퍼에, 카메라는 월드(≈베이스)에 **강체로 고정**되어 있으므로 $X$, $Z$는
모든 자세에서 불변이다. 자세 $i$에서 베이스→보드 경로는 두 가지다:

- 로봇을 거치는 경로: $^{b}T_{t,i} = {^{b}T_{g,i}}\,{^{g}T_{t}} = A_i X$
- 카메라를 거치는 경로: $^{b}T_{t,i} = {^{b}T_{c}}\,{^{c}T_{t,i}} = Z B_i$

두 경로는 같은 변환이므로

$$
A_i X = Z B_i \qquad \forall i
\tag{1}
$$

이것이 **AX=ZB** 문제다. $A$는 FK, $B$는 PnP 결과를 **그대로** 쓰면 되고,
풀고 나면 사용자가 원하는 두 관계 — ① 말단→체커보드 $X$, ② 베이스→카메라 $Z$ —
를 모두 얻는다.

> **AX=XB 와의 관계**: (1)의 두 자세 $i,j$를 조합하면
> $A_j^{-1}A_i X = X B_j^{-1}B_i$, 즉 $Z$가 소거된 AX=XB 가 된다.
> MG400 실습은 이 형태(OpenCV `calibrateHandEye`)로 $X$만 먼저 풀고 나머지를
> 복원했다. AX=ZB 폐형해는 소거 없이 전체 데이터를 한 번에 쓰므로 일반적으로 더 안정적.

## 2. 해의 유일성 (관측 가능성)

회전 부분만 보면 $R_{A,i} R_X = R_Z R_{B,i}$. 자세 쌍 $(i,j)$의 상대 회전
$R_{A,j}^{-1}R_{A,i}$ 축이 **서로 평행하지 않은 쌍이 2개 이상** 있으면 $R_X, R_Z$가
유일하게 결정되고, 이어서 병진도 선형 방정식으로 풀린다.

- **6DOF 로봇 (본 실습)**: 손목 롤/피치(J4·J5·J6)가 다양한 회전축을 만들어
  축퇴가 없다 → 게이지 고정 불필요, $X$ 6 + $Z$ 6 = **12 파라미터 전부 관측 가능**.
- **SCARA (MG400 실습)**: 모든 회전축이 $z$ 와 평행 → $z$ 방향 병진이 관측 불가라
  '자로 잰 카메라 높이'로 게이지를 고정해야 했다. 이 차이가 두 실습의 핵심 대비점.

## 3. 파이프라인

`robot_eye_calibration.py` 의 단계:

1. **데이터 수집** — 노미널 자세(보드가 카메라 정면) 주변 랜덤 관절 자세를
   만들고(시드 고정), 보드가 검출된 자세만 채택 ($N=18$). 직렬 체인이라 `qpos`
   직접 지정 + `mj_forward` 로 자세 생성 (MG400은 폐링크라 PD 제어로 정착시켜야 했음).
   J4/J6 롤 섭동을 크게 주어 **회전축 다양성**을 확보한다 (§2).
2. **내부 파라미터** — `cv2.calibrateCamera` (fx=fy, 접선왜곡 0, k3 고정).
3. **뷰별 보드 포즈** — calibrateCamera 의 extrinsic 이 곧 $B_i = {^{c}T_{t,i}}$.
4. **AX=ZB 폐형해** — `cv2.calibrateRobotWorldHandEye` (Shah / Li).
5. **비선형 정밀화** — 재투영 오차 최소화 (K, dist 도 동시 추정):

$$
\min_{X,Z,K,\mathrm{dist}}\;\sum_i \sum_k \left\| \pi\!\left(K,\,\mathrm{dist},\;
Z^{-1} A_i\, X\, {^{t}P_k}\right) - p_{ik} \right\|^2
$$

   ($Z^{-1}A_i X = {^{c}T_{t,i}}$ 이므로 보드 코너 $^{t}P_k$를 투영한 것.
   회전은 회전벡터로 파라미터화. $X$ 6 + $Z$ 6 + 내부 5(f, cx, cy, k1, k2) = 17
   파라미터, `scipy least_squares` LM. [2]의 K 는 로봇 FK 를 안 쓴 추정이라 잔여
   오차가 있고 X/Z 만 풀면 그 오차를 X/Z 가 흡수하므로 K 도 함께 푸는 편이 낫다.)

6. **GT 비교** — 시뮬레이션이므로 $X$, $Z$의 정답을 알고 있어 오차를 정량 검증.

## 4. OpenCV 규약 매핑 (실수하기 쉬운 부분)

`cv2.calibrateRobotWorldHandEye(R_world2cam, t_world2cam, R_base2gripper, t_base2gripper)`

OpenCV 의 원형은 "**보드(world)가 베이스에 고정 + 카메라가 손에**" 구성으로,
만족하는 항등식은 $^{c}T_{g} = {^{c}T_{w,i}}\,{^{w}T_{b}}\,{^{b}T_{g,i}}$ 이다.
우리 구성("보드가 손에 + 카메라가 베이스에 고정")은 그 **쌍대(dual)** — 문자만
치환하면 항등식이 우리의 $ {^{g}T_{t}} = {^{g}T_{b,i}}\,{^{b}T_{c}}\,{^{c}T_{t,i}}$
(= 식 (1)) 와 정확히 같은 구조다. 따라서 역할을 바꿔 넣는다:

| OpenCV 슬롯 | 우리가 넣는/받는 것 | 비고 |
|---|---|---|
| 입력 `world2cam` | $A_i^{-1} = ({^{b}T_{g,i}})^{-1}$ | FK 를 **역변환**해서 입력 |
| 입력 `base2gripper` | $B_i^{-1} = ({^{c}T_{t,i}})^{-1}$ | PnP 를 **역변환**해서 입력 |
| 반환 `base2world` | $Z = {^{b}T_{c}}$ | **그대로** 사용 |
| 반환 `gripper2cam` | $X = {^{g}T_{t}}$ | **그대로** 사용 |

(MG400 실습에서 `calibrateHandEye` 에 base2gripper 를 넣던 "트릭"과 같은 발상 —
함수가 가정하는 구성과 실제 구성이 쌍대 관계면 입출력의 역할을 맞바꿔 쓸 수 있다.)

## 5. 보드 좌표계의 라벨링 모호성 (GT 검증 시 주의)

`findChessboardCorners` 는 같은 물리적 코너 격자를 **4가지 방법**으로 라벨링할 수
있다: 시작 코너(대각 2곳) × 행 진행 방향. 이 중 2가지는 PnP 좌표계의 $z$축이
보드 **안쪽**(카메라 반대편)을 향한다 — 이미지 $x$(오른쪽)·$y$(아래쪽)에 대응하는
보드 위 방향의 외적이 카메라에서 멀어지는 쪽이기 때문이며, 실제로 흔히 나온다.
추정된 $X$ 자체는 어느 라벨링이든 식 (1)을 만족하므로 **캘리브레이션에는 문제가
없고**, GT 와 비교할 때만 4가지 후보 프레임을 모두 만들어 추정값과 가장 가까운
것을 채택한다 (`ground_truth()`, `pick_X_GT()`).

$Z$ 에는 이런 모호성이 없다 — 어떤 라벨링이 선택되든 $B_i$ 에 **동일한** 고정
변환이 곱해질 뿐이고, 이는 전부 보드 쪽 미지수 $X$ 로 흡수된다.

## 6. 카메라 배치와 조건수

처음에 카메라를 멀리(0.85 m) 두고 화각을 좁게(fovy 35°) 했더니, 폐형해는 정확한데
**정밀화가 오히려 해를 나쁘게 만드는** 현상이 나왔다. 좁은 화각에서는 원근 효과가
약해 $f$(초점거리)와 시선 방향 깊이가 거의 축퇴 — 재투영 오차의 골짜기가 평평해져
LM 이 $f$ 를 틀린 값으로 밀면서 $X, Z$ 로 보상한다. 카메라를 가깝게(0.7 m),
화각을 넓게(fovy 45°) 바꾸면 원근이 강해져 축퇴가 풀리고 정밀화가 제대로 동작한다.
실기에서도 **좁은 화각(망원) + 평면 타깃** 조합은 hand-eye 정밀도를 떨어뜨리는
대표적 요인이다.

## 7. 실행

```bash
cd 6dof_cali
python3 robot_eye_calibration.py            # 실시간 창 + mp4 기록
python3 robot_eye_calibration.py --headless # 창 없이 (mp4 기록은 동일)
```

출력 (`calib_output/`):

- `collection.mp4` — 수집 과정 영상 (overview + 외부 카메라)
- `reprojection_check.png` — 검출 코너(초록) vs 최종 해 재투영(빨강)
- `robot_eye_result.json` — $K$, dist, $X$, $Z$ (폐형해/정밀화/GT)

### 결과 예시 (seed=3, 18 뷰)

| 항목 | 회전 오차 | 병진 오차 |
|---|---|---|
| $X$ (Li 폐형해) | 0.021° | 0.18 mm |
| $Z$ (Li 폐형해) | 0.106° | 0.27 mm |
| $X$ (정밀화 후) | 0.049° | 0.16 mm |
| $Z$ (정밀화 후) | 0.076° | 0.53 mm |

재투영 RMS 0.60 px → 0.15 px, $f$ 오차 0.1 % → 0.03 %. 시뮬레이션이라 폐형해가
이미 매우 정확해 정밀화의 이득이 작지만, 실기에서는 렌즈 왜곡·조명·기구 유격
때문에 폐형해와 정밀화의 차이가 훨씬 크게 나타난다.

## 8. 로봇 모델 (`robot6dof.xml`)

프리미티브 geom 만으로 만든 간단한 6DOF 직렬 팔:

```
base(원통) ─ J1(요,z) ─ J2(어깨 피치,y) ─[상완 0.25 m]─ J3(팔꿈치 피치,y)
  ─[전완 0.20 m]─ J4(전완 롤,z) ─ J5(손목 피치,y) ─ J6(공구 롤,z) ─ 플랜지(link6)
```

- 체커보드(내부 코너 9×6, 한 칸 8 mm)는 플랜지에서 옆으로 오프셋 + x축 0.12 rad
  기울여 부착 → $X$ 가 자명하지 않음. 보드 법선은 공구 +z 근처라 외부 카메라를 향한다.
- 외부 카메라는 로봇 앞(+x) 스탠드 위 (0.7, 0, 0.48), fovy 45°. MuJoCo 카메라는
  $-z$ 시선이므로 OpenCV 프레임으로 변환할 때 $R_{cv} = R_{mj}\,\mathrm{diag}(1,-1,-1)$.
- 캘리브레이션은 운동학만 사용하므로 gravity=0, 액추에이터 없음.
